Tales ships as a single static Go binary. Linux and macOS are supported on both `amd64` and `arm64`. There are four installation paths; pick the one that fits where you are running Tales.

## Homebrew (macOS / Linux)

The fastest path on a developer machine. The [`tales-testing/homebrew-tap`](https://github.com/tales-testing/homebrew-tap) repository hosts the cask.

```bash
brew install --cask tales-testing/tap/tales
```

To upgrade later:

```bash
brew update
brew upgrade --cask tales
```

The cask installs the same release archive published on GitHub, so what you get locally is byte-identical to what runs in CI. On macOS the cask also runs a `postflight` hook that strips the `com.apple.quarantine` attribute from the binary, so the first `tales --version` does not trip Gatekeeper's "developer cannot be verified" prompt.

<Aside type="caution" title="Upgrading from the legacy formula">
  Releases up to **v0.1.0-rc.7** shipped a Homebrew **formula** at `Formula/tales.rb`. From **v0.1.0** onwards Tales is distributed as a **cask** at `Casks/tales.rb`. If you installed the formula before, uninstall it once before switching:

  ```bash
  brew uninstall tales 2>/dev/null || true
  brew install --cask tales-testing/tap/tales
  ```

  The transition is one-shot — `brew upgrade --cask tales` keeps working after that.
</Aside>

## GitHub Action (CI)

Use the [`tales-testing/setup-tales-action`](https://github.com/tales-testing/setup-tales-action) GitHub Action to pin and install Tales on a runner.

```yaml
- name: Install Tales
  uses: tales-testing/setup-tales-action@v1
  with:
    version: latest   # or a pinned tag like v0.1.0

- name: Run scenarios
  run: tales test ./e2e/pass --seed 1234 --parallel 8
```

The action caches the binary across runs and exposes `tales --version` to subsequent steps. See [CI integration](/docs/reports/ci-integration/) for full workflow recipes.

<Aside type="tip" title="Pin the version in CI">
  In production pipelines, prefer a pinned tag (`v0.1.0`) over `latest`. A pinned version means a red build is reproducible months later with the exact same binary.
</Aside>

## Pre-built binary

Download the release archive for your platform from [GitHub Releases](https://github.com/tales-testing/tales/releases) and drop it into `PATH`. The GoReleaser archive name is `tales_<version>_<os>_<arch>.tar.gz`, so set `VERSION` to a published tag from the releases page (without the leading `v`) before running the snippet.

<Tabs syncKey="os">
  <TabItem label="Linux (x86_64)">
    ```bash
    VERSION=0.1.0   # check the releases page for the latest tag
    curl -fsSL -o tales.tar.gz \
      "https://github.com/tales-testing/tales/releases/download/v${VERSION}/tales_${VERSION}_linux_x86_64.tar.gz"
    tar -xzf tales.tar.gz
    sudo install -m 0755 tales /usr/local/bin/tales
    tales --version
    ```
  </TabItem>

  <TabItem label="macOS">

    VERSION=0.1.0   # check the releases page for the latest tag
    curl -fsSL -o tales.tar.gz \

## From source

Go 1.26+ and a working `git` are required.

```bash
git clone https://github.com/tales-testing/tales
cd tales
make install   # installs to $GOBIN, falls back to $GOPATH/bin
```

For a one-shot install without a clone:

```bash
go install github.com/tales-testing/tales/cmd/tales@latest
```

## Homebrew tap layout

If you maintain your own fork, the cask lives at `Casks/tales.rb` in [`tales-testing/homebrew-tap`](https://github.com/tales-testing/homebrew-tap) — this is what GoReleaser writes on every stable release:

```ruby
cask "tales" do
  desc "Declarative integration and e2e testing with .tales files"
  homepage "https://taleslabs.org"
  version "0.1.0"

  binary "tales"

  on_macos do
    on_intel do
      url "https://github.com/tales-testing/tales/releases/download/v#{version}/tales_#{version}_darwin_x86_64.tar.gz",
        verified: "github.com/tales-testing/tales/"
      sha256 "REPLACE_WITH_ACTUAL_SHA256"
    end
    on_arm do
      url "https://github.com/tales-testing/tales/releases/download/v#{version}/tales_#{version}_darwin_arm64.tar.gz",
        verified: "github.com/tales-testing/tales/"
      sha256 "REPLACE_WITH_ACTUAL_SHA256"
    end
  end

  on_linux do
    on_intel do
      url "https://github.com/tales-testing/tales/releases/download/v#{version}/tales_#{version}_linux_x86_64.tar.gz",
        verified: "github.com/tales-testing/tales/"
      sha256 "REPLACE_WITH_ACTUAL_SHA256"
    end
    on_arm do
      url "https://github.com/tales-testing/tales/releases/download/v#{version}/tales_#{version}_linux_arm64.tar.gz",
        verified: "github.com/tales-testing/tales/"
      sha256 "REPLACE_WITH_ACTUAL_SHA256"
    end
  end

  postflight do
    if OS.mac?
      system_command "/usr/bin/xattr", args: ["-dr", "com.apple.quarantine", "#{staged_path}/tales"]
    end
  end
end
```

Notable bits:

- `binary "tales"` makes the cask install the binary out of the unpacked archive directly into Homebrew's `bin` prefix; no `def install` block needed.
- `url ... verified:` declares the trusted download domain, which `homebrew audit` requires when it differs from `homepage`.
- The `postflight` hook strips the macOS Gatekeeper quarantine bit so the binary runs without user interaction on a fresh machine.

GoReleaser writes this file automatically on every non-prerelease tag (`skip_upload: auto` on prereleases). Bumping `version` and the four `sha256`s by hand is only needed if you operate a separate fork without GoReleaser.
