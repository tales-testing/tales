What it is

A pragmatic DSL for testing real service flows.

Tales sits between low-level HTTP scripts and heavyweight acceptance frameworks. It gives backend and platform teams a compact way to describe multi-step API behavior with stable outputs between steps.

The first V1 focus is HTTP workflows: JSON requests, response expectations, captures, teardown, tags, parallel execution, and deterministic replay.

Why Tales

Designed for suites developers actually maintain.

Single binary

Build once, run locally or in CI without a runtime-heavy toolchain.

Readable scenarios

Use declarative HCL2 files to keep requests, assertions, and captures close together.

Deterministic data

Seeded generators make failures reproducible, even when suites run in parallel.

Workflow aware

Expose stable step outputs with capture and reuse them through result.*.

Clean teardown

Always run cleanup blocks and guard them with when = can(...) for partial failures.

CI reports

Emit human console output, JUnit XML, and JSONL events from the same run.

Workflow

From scenario file to reproducible report.

  1. 01

    Describe the journey

    Model a user or service flow with named steps, requests, expectations, and captures.

  2. 02

    Run with a seed

    Generate stable test data, filter by tag or scenario, and scale with scenario-level parallelism.

  3. 03

    Debug fast

    Use replay commands, console progress, and structured reports to turn failures into fixes.

Docs

A quick look at the DSL.

Tales scenarios are plain text files with a .tales extension. The DSL is explicit enough for code review and compact enough to keep test intent visible.

scenario step expect capture teardown
tests/blog.tales 
version = 1

config {
  base_url = env("BASE_URL", "http://localhost:1337")
}

generator "email" "user_email" {
  prefix = "test-"
  domain = "example.com"
}

scenario "Create a blog post" {
  tags = ["demo"]

  step "http" "create_user" {
    request {
      method = "POST"
      url    = "${config.base_url}/users"
      json = {
        email    = generate("user_email")
        password = "Passw0rd!"
      }
    }

    expect {
      status = 201
      json = {
        id    = is_string()
        email = request.json.email
      }
    }

    capture {
      id       = response.json.id
      email    = response.json.email
      password = request.json.password
    }
  }
}
Terminal 
# Validate every .tales file in a folder
tales validate ./e2e

# Run a deterministic suite
BASE_URL=http://localhost:1337 tales test ./e2e/pass --seed 1234 --parallel 4

# Focus one scenario
tales test ./e2e/pass --scenario "Create a blog post"

# Publish CI reports
tales test ./e2e/pass \
  --report-junit build/reports/e2e.junit.xml \
  --report-jsonl build/reports/e2e.jsonl

Install

Build from source today.

Tales is currently distributed from the repository. Clone it, build the binary, and add it to your local path or CI image.

Build 
git clone https://github.com/hyperxlab/tales.git
cd tales
make build
./build/tales validate ./e2e
./build/tales test ./e2e/pass --seed 1234

Reference

Core commands and matchers.

Commands

  • tales validate <path>
  • tales test <path> --seed 1234
  • tales test <path> --parallel 4
  • tales test <path> --tag smoke

Matchers

  • contains(value)
  • matches(regex)
  • is_string(), is_number(), is_bool()
  • one_of(values), can(expression)
